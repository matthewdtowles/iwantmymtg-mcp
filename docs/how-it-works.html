iwantmymtg-mcp: how it works

A personal walkthrough of the MCP server - what it is, the one path a request takes end to end, then every file drilled down in the order you'd hit it.

Repo: ~/projects/iwantmymtg-mcp · 59 tools · Node ≥20, TypeScript ESM, stdio transport

The gist, in one paragraph

This is a thin, typed adapter. It owns no data and no business logic. An MCP client (Claude Desktop, Claude Code, Cursor) launches dist/index.js as a subprocess and talks JSON-RPC to it over stdin/stdout. The server advertises 59 tools; when the model calls one, the server validates the arguments with a zod schema, turns them into an HTTP request against the public iwantmymtg.net REST API, and hands the JSON back as text. Everything real - the cards, the prices, your collection, the Premium gating - lives behind that API. If a tool touches your own data it sends your IWMM_API_KEY as a bearer token; the 11 catalog tools (cards, sets, prices) need no key at all.

The two design ideas worth holding onto: the API's OpenAPI spec is the source of truth for types (they're code-generated at build time, so a backend change that breaks a tool is a compile error), and every fact about a tool is structured data on the tool object, not prose - requiresAuth drives the docs, the description suffix, and a test, so the three can't drift apart.

One request, end to end

Say the model calls get_card with { setCode: "lea", setNumber: "161" }. This is the whole path:

  MCP client (Claude Desktop / Code / Cursor)
        │  spawns `node dist/index.js`, speaks JSON-RPC over stdio
        ▼
  index.ts                 process entry; catches fatal errors → stderr
        ▼
  server.ts                MCP Server + StdioServerTransport
        │
        ├─ ListTools  ──►  listTools()         reads the registry, zod → JSON Schema
        │
        └─ CallTool   ──►  handleCallTool(name, args)
                              │
                              │ 1. look up the tool     tools/index.ts   (toolsByName)
                              │ 2. validate args        tool.inputSchema.safeParse()
                              │ 3. run it               tool.handler(args)
                              │        │
                              │        ▼
                              │   tools/get-card.ts     apiClient.GET("/api/v1/cards/{setCode}/{setNumber}")
                              │        │
                              │        ▼
                              │   api-client.ts         onRequest:  UA, Accept, Bearer token if opted in
                              │        │                onResponse: !ok → throw ApiError
                              │        ▼
                              │   https://iwantmymtg.net/api/v1/...    ← the real system
                              │        │
                              │        ▼
                              │   unwrap(data, error)   returns data, rethrows non-HTTP failures
                              │
                              │ 4. shape the result     undefined → "OK" | string → as-is | object → JSON
                              └─ on throw:              error-formatter.ts → friendly text, isError: true
        ▼
  MCP client renders the text block to the model

Four places can end the call, and each has a distinct shape: unknown tool name, invalid arguments, a thrown ApiError, or a successful result. handleCallTool is exported separately from the request handler purely so those four branches are unit-testable without standing up a transport.

Drill down: every file

Ordered the way a request flows. Open one for the detail.

src/index.ts - process entry, 8 lines

A shebang (#!/usr/bin/env node, so npm can expose it as the iwantmymtg-mcp bin), a call to startServer(), and a catch that writes to stderr and exits 1.

The stderr detail is not cosmetic: stdout is the MCP transport. Anything you console.log anywhere in this codebase corrupts the JSON-RPC stream and the client disconnects. If you're ever debugging this thing, log to stderr.

src/server.ts - MCP protocol wiring

Three exports, and the split is deliberate - the first two are pure functions so tests can call them directly.

listTools() - builds the ListTools response

Maps the registry into what the client sees. Three things happen here that are worth knowing:

  • The auth suffix is derived. If requiresAuth is true the description gets "Requires IWMM_API_KEY." appended. Nobody types that into a description by hand, so it can't go stale when a tool's auth-ness changes.
  • zod → JSON Schema via zodToJsonSchema, using the default jsonSchema7 target. Not the openApi3 target: that one emits nullable: true, which strict clients reject on fields like update_price_alert's nullable thresholds.
  • Annotations. readOnly and destructive become MCP's readOnlyHint / destructiveHint, so a client can require confirmation before delete_deck instead of guessing from the description text.
handleCallTool(name, rawArgs) - runs one tool call

The four branches, in order:

  1. Unknown tool → isError: true, "Unknown tool: x".
  2. Invalid args → safeParse fails, the zod message goes back verbatim so the model can correct itself and retry.
  3. Handler throws → formatError(err), isError: true. Never crashes the process.
  4. Success → result is shaped: undefined (a 204) becomes the literal "OK", a string (CSV exports) passes through unencoded, anything else is JSON.stringify(..., null, 2).

That undefined → "OK" case matters: JSON.stringify(undefined) is undefined, which would produce a text content block with no text field - invalid MCP, and a confusing client-side failure.

startServer() - boots the SDK server

Constructs the SDK Server with name + version, declares the tools capability, registers the two request handlers, and connects a StdioServerTransport. Stdio is the only transport here - the streamable-HTTP variant of this server lives in the main web app at /mcp, not in this repo.

src/tools/types.ts - the tool contract (read this one first)

The most load-bearing 40 lines in the repo. Defines ToolDefinition<S>, generic over its own zod schema, and the defineTool factory.

interface ToolDefinition<S extends z.ZodTypeAny> {
  name: string;
  description: string;
  requiresAuth: boolean;
  readOnly?: boolean;      // → MCP readOnlyHint
  destructive?: boolean;   // → MCP destructiveHint
  inputSchema: S;
  handler: (input: z.output<S>) => Promise<unknown>;
}

Why generic: the handler's parameter type is inferred from the schema via z.output<S>. There are no hand-written handler input types to drift, and a schema/handler mismatch fails the build rather than surfacing as a runtime shape bug.

Why defineTool exists: it's an identity function that pins the generic at the definition site so TypeScript checks the handler against the schema there, but it returns the erased ToolDefinition. ToolDefinition is invariant in S (the handler parameter is contravariant), so the narrow form wouldn't be assignable into a single ToolDefinition[]. The cast goes through unknown because a direct one is rejected. Runtime object is unchanged.

Rule of thumb: never construct a tool object literally. Always go through defineTool, or you lose the schema↔handler check.

src/tools/index.ts - the registry

Imports every tool and puts them in one ordered tools: ToolDefinition[], grouped by comment (read-only → sell → buy-list → inventory → sealed → transactions → portfolio → decks → alerts → notifications). Then toolsByName, an Object.fromEntries lookup used by the dispatcher.

This array is the single source for four consumers: the ListTools response, the dispatcher, docs/TOOLS.md generation, and the tests. Adding a tool is two edits - define it in its group file, add it to this array - and then npm run gen:tools-doc, which CI enforces.

src/tools/schemas.ts - shared zod primitives

The building blocks defined once so they can't drift between tools: FORMATS / formatEnum (the canonical 11 formats, shared by search, sets, and decks), idParam (z.coerce.number().int().min(1) - coerces because path params arrive as strings), cardIdSchema (the internal card UUID), pageParam, limitParam (capped at the API's max of 100), and cardKey (the { setCode, setNumber } pair that addresses one printing).

zod schemas are immutable, so consumers freely chain .optional() / .describe() without mutating the shared base.

The shape of a tool - get_card, annotated 
export const getCardTool = defineTool({
  name: "get_card",
  requiresAuth: false,       // no Bearer token; also drives docs + description suffix
  readOnly: true,            // → MCP readOnlyHint
  description:
    "Look up a specific card printing by set code and collector number. ... " +
    "For broader catalog search use search_cards.",
  inputSchema: z.object(getCardInputSchema),
  handler: getCard,
});

Note the description's last clause. These are prompts, not docs - they're the only thing the model reads when choosing between 59 tools, so the good ones say when not to use this tool and which one to use instead. Same reason setNumber's .describe() explains it's a string because of suffixes like '12a': without that, models pass an int and get a 404.

Authenticated tools - the AUTH_HEADERS opt-in 
handler: async (input) => {
  const { data, error } = await apiClient.GET("/api/v1/inventory", {
    params: { query: input },
    headers: AUTH_HEADERS,        // ← the opt-in
  });
  return unwrap(data, error);
}

AUTH_HEADERS is the sentinel { "X-IWMM-Auth": "required" }. The client middleware swaps it for a real Authorization: Bearer header. Auth is opt-in per request, so a catalog tool physically cannot leak your key.

The flag and the header must agree: requiresAuth: true plus AUTH_HEADERS. A test asserts this invariant across the whole registry, because getting it half-right is the easy mistake - requiresAuth: true without the header means the call silently runs unauthenticated and 401s confusingly.

src/api-client.ts - the typed HTTP layer

Wraps openapi-fetch around the generated paths type, so apiClient.GET("/api/v1/cards/{setCode}/{setNumber}") is checked against the real spec - a typo'd path or a renamed param is a compile error, not a 404 in production.

authMiddleware - headers in, errors out

onRequest: sets User-Agent: iwantmymtg-mcp/<version> and Accept: application/json; if the request carries X-IWMM-Auth: required, deletes that sentinel and sets Authorization: Bearer <key> via requireApiKey() - which throws a message telling you exactly where to create a key if it's missing.

onResponse: any non-2xx becomes a thrown ApiError carrying status, body text, and the X-RateLimit-* headers. That's why handlers have no error handling: every HTTP failure is already an exception by the time unwrap sees it.

getClient() / the Proxy - lazy + memoized

The exported apiClient is a Proxy whose getter calls getClient(). Two constraints are being satisfied at once:

  • Lazy, so tests can stub globalThis.fetch after module load and so config.baseUrl is read at call time, not import time.
  • Memoized, so we don't rebuild the client and re-register middleware on every property access. The cache is keyed on baseUrl and rebuilds when it changes, which preserves the "config picked up at call time" contract.

fetch is indirected through globalThis.fetch at call time rather than captured, so stubbing still works even against a memoized client.

unwrap(data, error) - the last three lines

openapi-fetch returns { data, error }. HTTP failures were already thrown by the middleware, so error here only ever carries non-HTTP failures - network errors, DNS. Those get rethrown (normalized to an Error); otherwise data comes back.

src/config.ts - environment, read lazily

Two env vars, both exposed as getters rather than captured constants so callers and tests see live values instead of an import-time snapshot:

VarDefaultMeaning
IWMM_BASE_URLhttps://iwantmymtg.netPoint at localhost to develop against a local backend.
IWMM_API_KEYnoneiwm_live_…, from /user/api-keys. Only the 48 authenticated tools need it.

requireApiKey() is the accessor that throws when it's missing, and its message names the setting and the URL to create a key - the error surfaces to the model, which then tells you what to do.

src/error-formatter.ts - failures the model can act on

Turns an ApiError into text worth showing. The status-specific cases exist because each one has a different user action:

StatusMessageBecause
401"Not authenticated. Set IWMM_API_KEY…"Missing or bad key - a config fix.
402 / 403API detail + upgrade linkPremium gating - a subscription decision, not a bug.
429"Rate limit exceeded. Resets at …"Retry later; the reset time comes from the response header.
otherStatus + rate-limit remaining + API messageGeneric, but keeps the useful context.

extractApiMessage parses the body defensively - it handles a string message, an array of messages (NestJS validation pipes emit those; they get joined with "; "), and a string error. A non-JSON body falls through to undefined and the raw body is used.

src/generated/api-types.ts - do not edit

Generated by openapi-typescript from the live spec at https://iwantmymtg.net/api/openapi.json. It's checked in so builds work offline and so the diff is reviewable when the backend changes.

scripts/build-types.mjs runs as prebuild, so a plain npm run build regenerates it first. Offline or the API is down? IWMM_SKIP_TYPEGEN=1 npm run build uses the checked-in copy. Different backend? IWMM_OPENAPI_URL=….

This is the coupling to the web repo. When the API adds or changes an endpoint, rebuild here and the compiler tells you what broke. That's the whole point - it's why the tools have almost no defensive runtime checks.

src/version.ts - version, via createRequire

Reads version out of package.json. Uses createRequire(import.meta.url) because this is an ESM package and JSON import assertions are still awkward across Node versions.

Feeds two things: the User-Agent and the MCP server identity in the handshake. package.json sits at 0.0.0-dev - CI stamps the real version at publish, same convention as the web repo.

The four test files

  • api-client.test.ts - middleware behavior: headers set, auth sentinel swapped, ApiError thrown with rate-limit data.
  • error-formatter.test.ts - one case per status branch, plus the body-parsing shapes.
  • server.test.ts - the four handleCallTool branches and the listTools mapping.
  • tools.test.ts - registry invariants: unique names, and every requiresAuth tool actually sends AUTH_HEADERS.

Distribution

Four channels, all pointing at the same stdio server:

  • npm - npx -y iwantmymtg-mcp, via the bin entry.
  • MCPB bundle - manifest.json + scripts/build-mcpb.sh. Stages a production-only copy (npm ci --omit=dev), syncs the manifest version from package.json so they can't drift, packs with @anthropic-ai/mcpb. One-click install in Claude Desktop.
  • Smithery - smithery.yaml. Its configSchema declares iwmmApiKey as optional, which is the honest framing: the catalog tools work without one. commandFunction just shells out to npx.
  • MCP registry - server.json, keyed on the mcpName io.github.matthewdtowles/iwantmymtg-mcp.

Things worth remembering

  • stdout is sacred. Log to stderr or you break the transport.
  • Adding a tool is two edits plus a doc regen: define it with defineTool in its group file, add it to tools/index.ts, run npm run gen:tools-doc.
  • Descriptions are prompts. The model picks among 59 tools using nothing else. Say when to use a different tool.
  • Auth is opt-in per request, via AUTH_HEADERS. Keep requiresAuth and the header in sync - there's a test, but know why it's there.
  • No business logic belongs here. If a tool wants to compute something, that computation probably belongs behind the API where the web app and the mobile app get it too.
  • Backend changed? Rebuild - the generated types turn API drift into compile errors. That's the safety net this design buys.
